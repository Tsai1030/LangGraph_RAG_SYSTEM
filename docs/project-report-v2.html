營造業知識庫 RAG 系統

基於 LangGraph 狀態機的企業知識問答系統——整合 Hybrid 檢索、CRAG 自我修正閉環、多模態輸入（影像／語音）與智慧表單代填，將營造工程作業規範轉化為可對話、可產出文件的知識服務。

LangGraphFastAPINext.js 16 ChromaDBBM25 + RRFPostgreSQL GPT-5.4Gemini 3.5 FlashSSE Streaming

// 01 系統總覽

使用者以文字、圖片或語音提問，系統在單一 LangGraph 狀態圖內完成意圖判斷、混合檢索、品質評分與自我修正，最終以 SSE 逐 token 串流回答，並可直接產出可下載的 Word／Excel 表單文件。

設計核心原則

  • 單一狀態圖、共享 TypedDict State——所有節點讀寫同一個 GraphState，新功能（如 VLM 圖片）以純加法欄位擴充，舊 checkpoint 完全相容。
  • LLM 角色分工——主回答、評分、表單結構化、讀圖、語音轉錄各自獨立的模型設定，以 provider:model 字串在 .env 一行切換供應商。
  • 重資料不進 State——圖片 base64、音訊 bytes 永不寫入 checkpoint；State 只存輕量參照（路徑／id），LLM 呼叫時現場讀檔。
  • 每一個 LLM 判斷都有防護欄——結構化輸出 + 越界正規化 + 失敗 fallback，模型輸出異常時系統降級而非崩潰。

// 02 系統架構

// 04 Hybrid 檢索演算法

語意向量擅長同義改寫，BM25 擅長精確術語與編號（「010101」「動員開工」）。兩路各取 top-20，以 Reciprocal Rank Fusion 融合——不需調權重、對分數尺度不敏感。

hybrid_retrievalalgorithm
def RRF(d):  # k = 60    return Σ  1 / (k + rank_r(d) + 1)   for r in {vector, bm25}vector : text-embedding-3-small → ChromaDB top-20bm25   : jieba 分詞（營造領域自訂詞典）→ BM25Okapi top-20 (score > 0)fusion : 同時命中兩路的 chunk 自動加分 → 取前 8 進 context
CONCURRENCY

原子快照設計

BM25 索引與 corpus 綁成單一 tuple，retrieve() 取 local 快照後運算——admin 觸發重建（POST /admin/vector/refresh-bm25）瞬間換置新索引，進行中的查詢不會發生新舊索引錯位。KB 離線更新後免重啟。

DUAL-PATH

改寫後雙路融合

CRAG 改寫 query 後，retriever 同時用「原始 query」與「改寫 query」平行檢索，再做一次 RRF——改寫只是補充訊號，不會覆蓋使用者原意。向量檢索與 BM25 計算亦以 asyncio.gather 平行。

知識庫工程

  • 來源為營造作業規範 markdown 化文件，依章節階層切塊，metadata 保留 source_file / parent_h2 / parent_h3 / section_code / image_paths / tags。
  • ChromaDB 採版本化目錄（chroma_versions/v1…v5），.env 切 CHROMA_ACTIVE_VERSION 藍綠切換，重建知識庫不影響線上版本。
  • chunk 關聯的規範圖片以 markdown 圖片語法注入 context，LLM 可在回答中直接引用，前端帶認證渲染。

// 05 CRAG 自我修正閉環

Corrective RAG：檢索結果先經 LLM 評分，不足時改寫查詢重試——讓「使用者口語」與「規範文件用語」的落差由系統自己彌平，而不是直接拿差的 context 硬答。

STAGE機制防護欄
評分結構化輸出 {decision, reason, missing_information}，評全部 8 chunks × 400 字與 responder 實拿 context 對齊，避免「子集評不足、全文其實夠」的假陰性
改寫missing_information 作為改寫上下文，永遠以原始 query 為基礎防多次 retry 語意漂移；輸出 > 40 字或含「無法／不確定」等失敗訊號 → fallback
重試上限 2 次；改寫成功 → 雙路 RRF 重檢索 → 再評分fallback 回原 query 時直接跳過重複檢索走終端路由（相同 query 再檢索一次是純浪費）
出場sufficient 或達上限 → form 意圖過 form_structurer，其餘並行進 responder即使 insufficient 出場，仍以現有最佳 context 作答並標註依據
worst_case_costanalysis
# 最壞路徑成本（單輪）vision(1) + summarize(1) + intent(1)+ [retrieve → grade(1) → rewrite(1)] × 3 輪+ respond(1) + source_filter(1)   ≈ 9 次 LLM 呼叫# 控制手段：grader 用 mini 模型 · 死迴圈跳過 · 壓縮非每輪觸發

// 06 意圖分類與路由

捨棄 keyword fast-path（「我要規範的詳細說明」含「我要」就誤判索取檔案），改為每輪一次 LLM 結構化分類，prompt few-shot 教規則。多花 ≈300ms 換每輪正確路由。

INTENT路由觸發例
qa檢索 RAG 或直答「動員開工需要哪些初期計畫？」
static_form_download直達 responder + 下載卡片「給我動員開工檢核表」
static_form_fill填表 session 狀態機「我要填《開工檢核表》」
dynamic_form_generate檢索 → form_structurer「做一份新人安全是非題」
form_continuation帶推斷主題詞再檢索「再加 10 題」
dynamic_form_exportform_exporter（不打 LLM）「把剛剛的表匯出成 Excel」

越界正規化（LLM 防護欄）

  • static_form_* 但 target_form_id 不在候選 ∪ session → 降級 qa（防幻覺表單 id）。
  • form_continuation 但缺 prev_form_data 或主題詞 → 改 dynamic_form_generate。
  • dynamic_form_export 但根本沒有上一份表 → 降級 qa。
  • 候選表單若來自「歷史拼接」而非本輪 query 直接命中，qa 路徑不帶——避免問新主題時舊表單下載提示一直黏在回答結尾。

// 07 對話記憶與壓縮

CHECKPOINTER

跨輪持久化

LangGraph AsyncPostgresSaver 以 conversation_id 為 thread——messages、填表 session、上一輪表單／圖片參照全部自動跨輪保留，服務重啟對話不中斷。

COMPACTION

token 預算控制

歷史超過 8,000 tokens 時觸發：保留最近 8 則，更舊的由 LLM 壓成摘要（RemoveMessage 刪除原文），摘要同步寫入 app DB——歷史無限長，prompt 成本有上限。

CONTINUITY

prev_* 延續鏈

prev_form_data／prev_image_refs 仿「最近一份」語意：上一輪沒有就繼續往前找，避免中間插一句閒聊就斷鏈。「那張圖的第三欄是什麼？」「再加十題」都能正確接續。

// 08 多模態輸入

影像（VLM）——進 Graph 的前置節點

  • 上傳存磁碟、State 只放 {id, path, mime} 參照——base64 永不進 checkpoint。
  • vision_intake 用 Gemini 做客觀 OCR／描述（不答題），解析前 500 字併入 query 驅動檢索（全文稀釋 embedding 訊號），完整解析另存供 responder 用全文。
  • 生成時原圖再附給 responder 當輪訊息核對像素；多輪追問沿用上輪解析、不重跑讀圖、不污染新檢索。

語音（STT）——刻意不進 Graph

音訊只是問題的載體，轉成文字後就是純文字 query，沒有「上一段錄音」的延續語意——所以採方案 A：graph 外轉錄。前端 MediaRecorder 錄音 → POST /chat/transcribe → Gemini 原生音訊理解轉錄 → 文字填回輸入框讓使用者確認修正後才送出。Graph、State、所有節點零改動；轉錄錯誤不會污染檢索；音訊 bytes 不落磁碟。

image_path
upload → 磁碟 → vision_intake（graph 內）→ query enrich → 檢索與回答
audio_path
錄音 → transcribe（graph 外）→ 輸入框 → 使用者確認 → 以純文字進 graph

// 09 表單子系統

營造業的核心產出是「文件」。系統把表單拆成四種能力，全部收斂在同一個對話介面：

能力機制
靜態表單下載registry 比對 form_id → 原版 docx 模板直接下載。
AI 代填（多輪）schema 欄位分群 → 每輪只追問一組（含 AI 代寫欄位給使用者過目）→ session 狀態機 collecting／ready／completed → 寫入 docx 副本，下載 token 綁 conversation 所有權。
動態表單生成檢索規範 context → form role 結構化輸出 JSON → responder 以 markdown 表格呈現，支援「再加幾題」延續。
匯出上一份動態表單 → xlsx／csv 轉檔，純程式不打 LLM。

// 10 資安設計

面向實作
認證JWT access（記憶體）+ HttpOnly refresh cookie + token_version 撤銷；bcrypt(12)；Google OAuth 限公司網域。
限流LLM 端點 per-user key（JWT sub，Funnel 反代下 per-IP 無效）：stream 20／upload 30／transcribe 15 每分鐘；auth 端點 per-IP 3–10／分。
路徑防護所有檔案服務端點 Path.resolve() + is_relative_to（防 .. 穿越與兄弟目錄前綴繞過）；下載 token 驗 conversation 所有權。
上傳硬化分塊讀取超限即斷（防記憶體放大）；圖片 magic bytes 判型、以實際內容為準不信 client 宣告；音訊檔頭檢查記警告。
資訊洩漏內部錯誤只記 log、前端回通用訊息；404 不洩漏資源存在性；certs／DB dump 強制 gitignore。

營運機制

  • LangSmith tracing——每輪 9 個節點的延遲、token、重試率全鏈路可追。
  • 每日背景清理——lifespan 任務每 24h 清 30 天以上的上傳圖與產出表單，關機乾淨取消。
  • BM25 熱重建——admin 端點免重啟刷新索引；Chroma 換版本走藍綠 + PM2 重啟。
  • 失敗不靜默——回覆存檔、state 讀取、摘要入庫失敗均記 log；LLM 節點失敗降級不崩潰。
  • 容量保護——/chat/stream 全域 Semaphore(20)，滿載回 503 而非排隊雪崩。
